让 OpenClaw 更懂你：个人知识库的构建与连接

想让 AI 助手真正理解你的工作方式和知识体系？本文详解三种个人知识库构建方案，从云端笔记到本地部署，让 OpenClaw 成为你的第二大脑。

🧠 为什么你的 AI 助手还不够"懂你"？

你是否遇到过这样的场景：

  • 问助手"上次那个项目文档放哪了"，它一脸茫然
  • 想让助手基于你的笔记写总结，却发现它根本访问不了
  • 每次对话都要重新解释背景信息，仿佛在和失忆症患者聊天

问题不在于 AI 不够聪明，而在于它没有接入你的知识体系。

OpenClaw 作为你的个人智能体平台，天然具备连接各类知识源的能力。但如何选择合适的方案，让 OpenClaw 真正"记住"你的知识？

本文提供三种经过实战验证的方案，从轻量级云端同步到完整的本地 RAG 系统，帮你找到最适合自己的路径。

方案一：云端笔记系统对接 🌐

适用场景

  • 需要多设备实时同步
  • 不介意数据存储在云端

以 Memos 为例的对接步骤

1. 部署 Memos（可选自托管）

# Docker 快速部署
docker run -d \
  --name memos \
  -p 5230:5230 \
  -v ~/.memos/:/var/opt/memos \
  neosmemo/memos:latest

2. 在 OpenClaw 中配置技能

创建或编辑技能配置文件：

# ~/.openclaw/skills/memos-connector/config.yaml
name: memos-connector
version: 1.0.0
triggers:
  - "查询我的笔记"
  - "搜索 memos"
  - "找到我写的关于"

endpoints:
  base_url: http://your-memos-instance:5230
  auth_token: ${MEMOS_TOKEN}

3. 编写检索技能

// skills/memos-connector/SKILL.js
async function searchMemos(query, limit = 5) {
  const response = await fetch(`${config.base_url}/api/memo`, {
    headers: {
      'Authorization': `Bearer ${config.authToken}`
    }
  });
  
  const memos = await response.json();
  const filtered = memos.filter(m => 
    m.content.toLowerCase().includes(query.toLowerCase())
  );
  
  return filtered.slice(0, limit).map(m => ({
    id: m.id,
    content: m.content,
    createdAt: m.createdAt,
    tags: m.tags
  }));
}

4. 测试查询

# 在 OpenClaw 中测试
openclaw skill test memos-connector --query "项目会议记录"

Notion 对接要点

Notion 提供更完善的 API，但需要 OAuth 认证：

  1. 在 Notion Developers 创建 Integration
  2. 获取 integration_token
  3. 在 OpenClaw 配置中设置环境变量
  4. 使用 @notionhq/client 库进行查询
const { Client } = require('@notionhq/client');

const notion = new Client({ auth: process.env.NOTION_TOKEN });

async function searchDatabase(query) {
  const response = await notion.databases.query({
    database_id: process.env.NOTION_DATABASE_ID,
    filter: {
      property: 'Name',
      rich_text: {
        contains: query
      }
    }
  });
  return response.results;
}

⚠️ 注意事项

  • API 速率限制：Notion 免费版 3 次/秒，付费版 5 次/秒
  • 数据缓存：建议在 OpenClaw 侧实现缓存层，减少 API 调用
  • 权限管理：仅授权必要的数据库/页面访问权限

方案二：AI 笔记平台深度集成 🤖

适用场景

  • 已有大量笔记积累
  • 需要 AI 辅助理解和总结
  • 接受云端 AI 处理

Google NotebookLM 方案

NotebookLM 是 Google 推出的 AI 笔记平台，支持上传文档并自动建立知识关联。

对接思路

虽然 NotebookLM 没有公开 API，但可以通过以下方式间接集成：

  1. 导出笔记为 Markdown
  2. 存入 OpenClaw 可访问的存储
  3. 使用嵌入模型建立索引
# 批量导出 NotebookLM 笔记（需手动操作）
# 然后放入 OpenClaw 知识库目录
mkdir -p ~/.openclaw/knowledge/notebooklm
cp ~/Downloads/notebooklm-export/*.md ~/.openclaw/knowledge/notebooklm/

在 OpenClaw 中建立索引

# ~/.openclaw/config/knowledge-base.yaml
sources:
  - name: notebooklm-export
    type: markdown
    path: ~/.openclaw/knowledge/notebooklm
    embedding_model: text-embedding-3-small
    chunk_size: 500
    chunk_overlap: 50

完整部署步骤

1. 准备知识库目录

# 创建知识库目录结构
mkdir -p ~/knowledge/{obsidian,documents,exports}

# 同步 Obsidian 笔记（如果使用）
rsync -av ~/Obsidian-Vault/ ~/knowledge/obsidian/

2. 部署本地嵌入模型

使用 Ollama 运行轻量级嵌入模型：

# 安装 Ollama
curl -fsSL https://ollama.ai/install.sh | sh

# 拉取嵌入模型
ollama pull nomic-embed-text

# 验证运行
ollama run nomic-embed-text "测试文本"

3. 部署向量数据库

使用 Chroma（轻量级，适合个人使用）：

# 创建 Chroma 服务目录
mkdir -p ~/.chroma/data

# 使用 Docker 部署
docker run -d \
  --name chroma \
  -p 8000:8000 \
  -v ~/.chroma/data:/chroma/chroma \
  chromadb/chroma:latest

4. 创建 OpenClaw 知识索引技能

# ~/.openclaw/skills/local-kb/config.yaml
name: local-kb
version: 1.0.0
description: 本地知识库 RAG 检索

dependencies:
  - chromadb
  - sentence-transformers

config:
  chroma_host: http://localhost:8000
  chroma_collection: personal-knowledge
  embedding_model: nomic-embed-text
  knowledge_paths:
    - ~/knowledge/obsidian
    - ~/knowledge/documents

5. 编写索引构建脚本

#!/usr/bin/env python3
# skills/local-kb/scripts/build-index.py

import chromadb
from pathlib import Path
import hashlib

def generate_embedding(text):
    """调用 Ollama 生成嵌入向量"""
    import requests
    response = requests.post(
        'http://localhost:11434/api/embeddings',
        json={
            'model': 'nomic-embed-text',
            'prompt': text
        }
    )
    return response.json()['embedding']

def build_index():
    # 连接 Chroma
    client = chromadb.HttpClient(host='localhost', port=8000)
    collection = client.get_or_create_collection('personal-knowledge')
    
    # 遍历知识库
    knowledge_dir = Path.home() / 'knowledge'
    doc_id = 0
    
    for md_file in knowledge_dir.rglob('*.md'):
        content = md_file.read_text()
        
        # 分块（简单按段落）
        chunks = content.split('\n\n')
        
        for chunk in chunks:
            if len(chunk.strip()) < 50:  # 跳过太短的块
                continue
                
            embedding = generate_embedding(chunk)
            
            collection.add(
                ids=[f"doc-{doc_id}"],
                embeddings=[embedding],
                documents=[chunk],
                metadatas=[{
                    'source': str(md_file),
                    'created_at': md_file.stat().st_mtime
                }]
            )
            doc_id += 1
    
    print(f"✅ 索引构建完成，共 {doc_id} 个文档块")

if __name__ == '__main__':
    build_index()

// skills/local-kb/SKILL.md

#### 7. 定时更新索引

在 OpenClaw 中配置定时任务：

```yaml
# ~/.openclaw/config/cron.yaml
tasks:
  - id: kb-index-update
    schedule: "0 2 * * *"  # 每天凌晨 2 点
    command: "openclaw skill call local-kb rebuild-index"
    notify: true

性能优化建议

优化项建议配置
分块大小300-500 tokens
分块重叠50-100 tokens
检索数量返回 3-5 个最相关片段

索引更新增量更新优于全量重建

分层存储策略：

层级存储位置更新频率检索优先级
L1: 热数据本地向量库每日最高
L2: 温数据云端笔记实时中等
L3: 冷数据归档存储按需最低

⚠️ 常见坑与解决方案

1. 检索结果不相关

原因：分块策略不当或嵌入模型选择不佳

解决：

  • 调整分块大小（300-500 tokens 通常最佳）
  • 尝试不同嵌入模型（nomic-embed-text vs text-embedding-3-small）
  • 添加元数据过滤（按时间、标签等）

2. 响应速度慢

原因：每次查询都实时生成嵌入

解决：

  • 预计算并缓存常用查询的嵌入
  • 使用更快的嵌入模型（如 all-MiniLM-L6-v2）
  • 增加向量数据库的索引优化

3. 知识更新不同步

原因：笔记修改后索引未更新

解决：

  • 配置定时重建任务（每日凌晨）
  • 实现文件监听自动触发更新
  • 提供手动刷新命令
